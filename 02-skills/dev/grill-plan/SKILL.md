# Grill Plan v2

在写任何代码之前，通过结构化的审问—结晶—验证循环消除需求歧义，直到可以安全动手。

核心原则：**歧义是 AI 编码失败的根本原因。不要猜，要问；不要开始，要收敛。**

---

## 触发条件

用户说"帮我规划这个功能"、"我想做 X"、"grill me"、"审问需求"，或准备开始任何非平凡的实现任务时。

---

## 执行流程

```
Interview → Seed → Execute → Evaluate
    ↑                            ↓
    └────── Evolutionary Loop ───┘
            (歧义分数 > 0.2 时循环)
```

---

### Phase 1：Interview（审问）

目标：把需求的歧义分数从未知降到 ≤ 0.2。

**歧义分数定义（自评，0.0 = 完全清晰，1.0 = 完全模糊）：**
- 0.8–1.0：不知道要做什么，问题根本没说清楚
- 0.5–0.7：大方向明确，但关键细节缺失
- 0.2–0.4：基本清楚，有少量边界模糊
- 0.0–0.1：完全清晰，可以立即实施

**规则：每轮最多问 3 个问题，按优先级排序，最重要的先问。**

#### 问题库——按维度选取最相关的：

**目标维度：**
- 这个功能要解决什么核心问题？谁在什么场景下用？
- 成功的标准是什么？什么输入对应什么输出算"完成"？
- 有没有已有的类似功能可以参考或复用？

**约束维度：**
- 有没有不能碰的文件、接口、依赖、或架构决策？
- 性能要求？并发量？数据规模？响应时间上限？
- 需要向后兼容吗？会影响哪些现有用户或系统？

**边界维度：**
- 这次明确做什么，明确不做什么？
- 有没有"看起来简单实际很难"的隐藏复杂度？
- 失败时的行为是什么？错误处理和降级策略？

**验证维度：**
- 怎么测试？单测够还是需要集成测试 / E2E？
- 谁来验收？验收标准是什么？

#### 每轮审问结束后输出歧义评分：

```
歧义分数：X.X / 1.0
剩余不确定点：[列出还不清楚的地方]
```

若歧义分数 > 0.2，进入下一轮 Interview。  
若歧义分数 ≤ 0.2，进入 Phase 2。

---

### Phase 2：Seed（结晶）

歧义收敛后，输出一份"需求结晶"——把所有审问结论压缩成最简洁、最精确的规格：

```markdown
## 需求结晶：[任务名称]

### 根问题（一句话）
[用户真正要解决的是什么，不是表面诉求]

### 输入 / 输出契约
- 输入：[精确描述]
- 输出：[精确描述，包括成功和失败两种情况]

### 做 / 不做边界
做：
- [具体条目]
不做（本次）：
- [明确排除的内容]

### 关键约束
- [不可突破的限制]

### 验收标准
- [具体可测的条件，而非模糊描述]
```

确认："以上需求结晶准确吗？有需要修正的地方吗？"

若用户修正 → 更新结晶 → 重新评估歧义分数。  
若用户确认 → 进入 Phase 3。

---

### Phase 3：Execute Plan（实施计划）

基于结晶输出结构化实施计划：

```markdown
## 实施计划：[任务名称]

### 实施步骤
| 步骤 | 操作 | 预计时间 | 验证方式 |
|------|------|---------|---------|
| 1 | [具体操作] | [时长] | [如何确认完成] |
| 2 | [具体操作] | [时长] | [如何确认完成] |
| 3 | [具体操作] | [时长] | [如何确认完成] |

### 风险与缓解
| 风险 | 概率 | 缓解方式 |
|------|------|---------|
| [风险描述] | 高/中/低 | [应对策略] |

### 依赖与前提
- [需要先完成或确认的内容]

### 已知未知（Known Unknowns）
- [不确定的地方，标注 ⚠ 需在实施中确认]
```

---

### Phase 4：Evaluate（评估循环）

**每个主要步骤完成后**，执行轻量评估：

```
实施评估 — 步骤 N
✓ 完成条目：[已验证的内容]
✗ 偏差：[与计划的差异]
? 新发现的未知：[实施中冒出的新问题]
歧义分数（更新）：X.X
```

若评估中发现偏差或新未知导致歧义分数重新升高 > 0.2：
→ 触发**演化循环**：返回 Phase 1，针对新问题重新 Interview，更新 Seed，调整计划。

若所有步骤评估通过：
→ 输出**完成回顾**：

```markdown
## 完成回顾

### 交付物
- [实际完成的内容]

### 与原计划的差异
- [偏差说明，若无则写"无偏差"]

### 经验提炼
- [这次发现的隐藏复杂度 / 值得下次注意的点]

### 遗留项
- [未完成的内容及原因，建议后续处理方式]
```

---

## 快速参考

| 场景 | 动作 |
|------|------|
| 需求一句话说不清楚 | 立即进 Phase 1，不要开始写代码 |
| 用户说"你来定" | 给出有理由的推荐，但标注假设条件 |
| 实施中发现新问题 | 触发演化循环，更新 Seed 再继续 |
| 用户催促跳过审问 | 说明风险，但尊重决定，在计划中标注 ⚠ |
| 歧义分数卡在 0.3 无法降低 | 接受残余不确定性，在计划中明确标注已知未知 |

---

## 注意事项

- 审问阶段**不给解决方案**，专注理解问题
- 每个步骤必须有对应的验证方式，不接受"完成了就是完成了"
- 演化循环不是失败，是设计的一部分——真实需求在实施中会涌现
- 歧义分数是内部工具，输出给用户时用自然语言描述，不要显示数字
