# Diagnose

系统化排障流程。用结构化方法代替随机试错。

核心原则：**先观察，再假设，再验证，不要跳步。**

## 触发条件

用户说"这个报错了"、"帮我调试"、"diagnose"，或粘贴了错误信息/异常行为描述。

## 执行流程

### Step 1：症状收集

先问清楚（如果上下文不足）：

1. **症状是什么？** — 具体的错误信息、异常行为，越精确越好
2. **期望行为是什么？** — 正确情况下应该发生什么
3. **最近改了什么？** — 上次正常运行到现在，做了哪些变更？
4. **可以稳定复现吗？** — 每次都出现 / 偶发 / 特定条件下出现？
5. **环境信息** — OS、运行时版本、相关配置（只要和问题相关的）

### Step 2：假设生成

基于症状，列出 2-4 个最可能的假设（从最高概率到最低概率排序）：

```
假设 1（最可能）：[具体假设]
  — 支持证据：[为什么认为是这个]
  — 验证方法：[用什么命令/检查来确认或排除]

假设 2：[具体假设]
  — 支持证据：...
  — 验证方法：...
```

### Step 3：验证循环

按优先级验证假设。每次验证只做一件事，看结果后再决定下一步。

```
验证 → 结果 → [确认假设 → 修复] 或 [排除假设 → 下一个]
```

**验证工具优先级：**
1. 读取日志 / 错误输出（最快，零侵入）
2. 检查配置文件和环境变量
3. 最小复现用例（隔离问题）
4. 添加临时 debug 日志
5. 逐步回退变更（git bisect 等）

### Step 4：修复与验证

找到根因后：
1. 实施最小化修复（不要顺手改其他东西）
2. 验证修复解决了原问题
3. 验证没有引入新问题（跑已有测试）
4. 如果问题有复发风险，建议添加测试用例

### Step 5：复盘（可选）

```markdown
## 排障复盘

**根因：** [一句话描述]
**耗时：** [分钟]
**为什么难发现：** [误导因素]
**预防措施：** [怎么避免下次再发生]
**值得固化为测试的场景：** [是/否，如果是，描述测试用例]
```

---

## 常见陷阱

- 不要在没有验证假设前就开始改代码
- 不要同时改多个地方（会导致不知道是哪个改动起了作用）
- 不要忽略错误信息的细节（"第 47 行" 比 "某个地方" 精确得多）
- 不要假设最新的改动就是根因（可能是触发了之前潜伏的 bug）
