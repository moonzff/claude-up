# Skill: sp-tdd
> 适配：Node.js / TypeScript 项目 · 版本：v1.0

## 定位

测试驱动开发（TDD）流程辅助。确保先写测试、再写实现，避免"先写代码、后补测试"的坏习惯。

适用场景：
- 新功能开发（测试即 Spec）
- Bug 修复（先复现 Bug 的测试，再修代码）
- 重构（用现有测试保护行为不变）

---

## 执行流程（Red → Green → Refactor）

### Phase 1 · Red — 写失败的测试

**目标**：用测试描述期望行为，测试必须先失败。

1. 确认测试框架（Jest / Vitest / Mocha）
2. 为一个功能点写最小可能的测试
3. 运行测试，**确认它失败**（Red）
4. 测试命名格式：`it('should <行为> when <条件>')`

```typescript
// 示例：先写测试
it('should return 0 for empty array', () => {
  expect(sum([])).toBe(0);
});
// 运行 → 失败 ✓（函数还不存在）
```

### Phase 2 · Green — 写最少代码让测试通过

**目标**：只写让测试通过所必需的代码，不多写。

1. 实现功能，不考虑设计质量
2. 运行测试，**确认它通过**（Green）
3. 如果实现超过 10 行，考虑是否测试粒度太粗

```typescript
// 最小实现
function sum(arr: number[]): number {
  return arr.reduce((a, b) => a + b, 0);
}
// 运行 → 通过 ✓
```

### Phase 3 · Refactor — 重构，测试保护行为

**目标**：改善代码设计，测试全程保持绿色。

重构检查项：
```
□ 提取重复逻辑为函数
□ 改善命名
□ 消除魔法值
□ 优化性能（如有必要）
□ 每次改动后立刻运行测试
```

### Phase 4 · 循环

一个测试周期（Red → Green → Refactor）完成后：
- 还有更多行为需要覆盖？回到 Phase 1
- 所有核心路径覆盖完？进入验收

---

## 测试覆盖检查

```
□ Happy path（正常输入）
□ Edge cases（空值、零、最大值、边界）
□ Error cases（无效输入、异常抛出）
□ 异步操作（Promise reject、timeout）
```

---

## 与项目框架的适配

**Jest（推荐）**：
```bash
npx jest --watch          # 监听模式，实时反馈
npx jest --coverage       # 覆盖率报告
```

**测试文件结构**：
```
src/
  utils/
    sum.ts
    sum.test.ts    # 与实现文件同目录
```

---

## 输出物

TDD 完成后，确认：
1. 所有测试绿色通过
2. 覆盖率报告中核心逻辑 ≥ 80%
3. 测试命名清晰，可作为行为文档读

---

## 原则

- **测试是 Spec，不是负担**：测试名描述业务行为，而不是实现细节
- **小步快跑**：每次 Red-Green 周期不超过 10-15 分钟
- **不测私有实现**：只测公共接口，内部实现可以随意重构
- **失败要有意义**：测试失败时，错误信息要清楚说明"期望什么、得到什么"
