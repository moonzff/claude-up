# Skill: sp-brainstorming
> 来源：obra/superpowers · 适配：Moonz 开发场景 · 版本：v1.0

## 定位

在写任何一行代码之前，先把"我们要做什么"搞清楚。
防止直接跳进实现，结果做完了才发现方向不对。

**接在 grill-plan 之后使用**：
- `grill-plan`：澄清需求，消除歧义，确认 Seed（做什么）
- `sp-brainstorming`：探索设计方案，确认架构（怎么做）
- `sp-writing-plans`：把确认的设计拆成可执行任务

---

## 执行流程

### Phase 1 · 提问澄清（一次一问）

- 每条消息**只问一个问题**，不轮番连问
- 优先用选择题（A/B/C），减少用户思考负担
- 持续提问直到能回答：目的是什么？约束是什么？成功标准是什么？

**核心问题方向**（不必全问，按需选取）：
1. 这个功能服务于谁？主要使用场景是？
2. 性能/规模要求：并发量、数据量、响应时间？
3. 与现有系统的边界：复用什么、新建什么？
4. 最重要的约束：时间、技术栈、兼容性？
5. 成功的样子是什么？怎么验证做好了？

### Phase 2 · 方案探索

理解需求后，提出 **2-3 个设计方案**，每个方案说明：
- 核心思路（1-2 句话）
- 主要优势
- 主要代价/风险
- 推荐指数（推荐 / 备选 / 不推荐 + 原因）

### Phase 3 · 设计分段展示

用户选定方案后，**分段展示设计**，每段结束询问"这部分 OK 吗？"再继续。

标准设计段落：
```
[架构概览] → 确认 →
[核心组件/模块划分] → 确认 →
[数据流 / 接口设计] → 确认 →
[错误处理策略] → 确认 →
[测试策略] → 确认
```

每段篇幅：简单场景 2-5 句话，复杂场景最多 200-300 字。

### Phase 4 · Spec 输出

所有段落确认后，输出完整 Spec 文档：
- 保存至 `docs/specs/YYYY-MM-DD-<feature>.md`（项目目录下）
- 包含：背景、目标、方案选择理由、完整设计、验收标准

---

## 原则

- **YAGNI**：需求里没有的功能，坚决不加
- **拆分大项目**：如果范围太大，先拆成独立子项目，每个子项目走独立的 brainstorm → plan → implement 循环
- **可以回头**：执行中发现设计有问题，回到此 Skill 重新 brainstorm，不要硬撑
- **Spec 是合同**：写完 Spec 后，后续 plan 和 execute 都以 Spec 为基准
