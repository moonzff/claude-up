# Skill: sp-writing-plans
> 来源：obra/superpowers · 适配：Moonz 开发场景 · 版本：v1.0

## 定位

把 Spec（设计文档）转换成可执行的实施计划。
计划颗粒度：每个任务 **2-5 分钟**可完成，包含具体文件路径、代码片段、验证步骤。

**前置条件**：已有 Spec 文档（来自 sp-brainstorming 或 grill-plan v2）

---

## 计划文件格式

保存至：`docs/superpowers/plans/YYYY-MM-DD-<feature>.md`

```markdown
# 实施计划：<功能名>
> Spec: [link]  | 日期: YYYY-MM-DD | 状态: 待执行

## 上下文速览
<3-5 句话：项目背景、技术栈、关键约束>

## 任务列表

### Task 1: <动词 + 名词，如"创建 User 数据模型">
**估时**: 3 分钟
**文件**:
- 新建: `src/models/user.ts`
- 修改: `src/db/schema.ts`

**实现**:
```typescript
// src/models/user.ts
export interface User {
  id: string
  email: string
  // ...
}
```

**验证**:
- [ ] `npx ts-check src/models/user.ts` 无报错
- [ ] 已有测试仍然通过

---

### Task 2: ...
```

## 写计划的规则

**任务颗粒度**：
- 每个任务专注单一职责
- 预估 2-5 分钟完成（不是小时）
- 如果超过 5 分钟，继续拆分

**必须包含**：
- 精确的文件路径（相对项目根目录）
- 关键代码片段（不是伪代码，是可直接用的代码）
- 具体可验证的检查步骤（命令或断言，不是"测试一下"）
- 依赖关系（如果 Task B 依赖 Task A，要标注）

**禁止**：
- 含糊步骤（"实现逻辑"、"添加功能"）
- 占位符（TBD、TODO、待定）
- 引用未定义的类型或函数
- 一个任务做多件事

---

## 自检清单（写完计划后执行）

```
□ 逐条对照 Spec，每个需求都有对应的 Task（无遗漏）
□ 搜索计划文件，无 "TBD"、"TODO"、"待定" 等占位词
□ 所有文件路径都是真实存在或将要创建的
□ 所有引用的类型/函数都有定义或将被创建
□ Task 顺序合理，依赖关系正确
□ 估时合理（单任务 2-5 min，全计划不超过半天）
```

---

## 原则

- **为零上下文的人写**：假设执行者对代码库一无所知，但技术能力强
- **DRY + YAGNI**：计划里不写 Spec 没有的内容
- **TDD 优先**：每个 Task 的验证步骤优先用测试，而不是手动检查
- **频繁提交**：每个 Task 完成后就是一个独立提交的节点
