# Skill: code-review
> 适配：Moonz 全栈项目（Node.js / TypeScript / Python） · 版本：v1.0

## 定位

对代码变更做系统性审查，不遗漏任何关键维度。
防止 review 流于表面（只看语法、不看设计）。

---

## Review 框架（六维度）

### 1. 正确性
```
□ 核心逻辑是否符合需求/Spec？
□ 边界条件是否处理（空值、零、最大值、并发）？
□ 异步操作是否正确（async/await、Promise 链、错误冒泡）？
□ 类型系统是否正确使用（TS 无 any 滥用）？
```

### 2. 安全性（必查）
```
□ 用户输入是否经过验证和 sanitize？
□ SQL / NoSQL 查询是否防注入（参数化查询）？
□ 敏感数据是否在日志/响应中泄露？
□ 权限校验是否在正确位置（不只在前端）？
□ 依赖包是否有已知漏洞（npm audit）？
```

### 3. 性能
```
□ 是否有 N+1 查询问题？
□ 循环内是否有不必要的 DB/API 调用？
□ 大数据集是否用了分页？
□ 是否有不必要的同步阻塞操作？
```

### 4. 可维护性
```
□ 函数/方法是否单一职责？
□ 命名是否清晰自解释？
□ 魔法值是否提取为常量？
□ 注释是否说明"为什么"而不是"是什么"？
□ 重复代码是否超过 3 次（可抽取）？
```

### 5. 测试覆盖
```
□ 核心业务逻辑是否有单元测试？
□ 关键 API 是否有集成测试？
□ 测试是否测了正常路径 + 至少一个异常路径？
□ 测试命名是否描述了测试意图？
```

### 6. 架构一致性
```
□ 是否遵循项目现有的分层结构（controller/service/model）？
□ 是否与项目的错误处理约定一致？
□ 是否引入了不必要的新依赖？
□ 是否与 API 约定一致（命名、响应格式、状态码）？
```

---

## 输出格式

```markdown
## Code Review · <文件/PR 名>

### 必须修复（blockers）
- [严重] xxx：原因 + 建议修法

### 建议改进（non-blockers）
- [建议] xxx：原因

### 做得好的地方
- xxx

### 总体评价
通过 / 需要修改后重审 / 不通过
```

---

## 原则

- **读代码，不读说明**：不依赖作者的描述，只看代码本身
- **blocker 要有理由**：每个必须修复的问题都要说清楚为什么严重
- **肯定好的部分**：只挑毛病的 review 是差的 review
- **建议要具体**：不写"这里应该更好"，写"建议改成 X，原因是 Y"
