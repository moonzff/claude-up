# Skill: verification-before-completion
> 来源：jnMetaCode/superpowers-zh 方法论 · 通用 · 版本：v1.0

## 定位

**证据先行**——在声称任何任务"完成"之前，必须先跑验证、看到证据，再汇报结果。

防止的问题：
- "应该好了"式的完成声明（没有实际验证）
- 实现了代码但没跑测试
- 修了 Bug 但没确认复现场景消失
- 部署了功能但没检查线上是否正常

**嵌入点**：在 `sp-executing-plans` 每个任务结束时、`systematic-debugging` Phase 4 结束时自动触发本 Skill 的检查逻辑。

---

## 完成前验证清单

### 通用检查（每次必做）

```
□ 我有没有实际运行代码，而不是只写了代码？
□ 我看到的输出/行为符合预期吗？（截图/日志/命令输出作为证据）
□ 如果有测试，测试是绿色的吗？
□ 如果任务有明确的验收标准，每一条都检查过了吗？
```

### 开发任务验证

```
□ 单元测试：npx jest / npm test 输出 → 全绿
□ 功能验证：在本地用实际请求/操作触发了目标功能
□ 边界情况：至少测试了一个异常输入/边界值
□ 相关功能回归：改动没有破坏周边功能
```

### Bug 修复验证

```
□ 能复现 Bug 的测试用例：修复前失败，修复后通过
□ 原始报错场景：手动走一遍，确认错误消失
□ 修复不是掩盖：根因被解决，不是错误被 catch 住了
□ 同类问题：同一模式在其他地方不再存在
```

### 部署/配置变更验证

```
□ 变更前备份或有回滚方案
□ 变更后健康检查通过
□ 日志中没有新增 ERROR
□ 核心功能冒烟测试：手动走一遍主要流程
```

---

## 汇报格式

声称完成时，必须包含验证证据：

```
✅ 任务完成：<任务名>

验证证据：
- 运行：<命令>
- 输出：<实际输出片段>
- 结论：<符合/不符合预期，以及原因>

如有遗留：
- ⚠️ <未验证的部分> — 原因：<为什么跳过>
```

**不接受的汇报格式**：
- "应该没问题了"
- "代码写好了"
- "我觉得可以了"
- "逻辑上是对的"

---

## 在 sp-executing-plans 中的嵌入

每个任务执行完毕，在标记 `[x]` 之前，先过本清单：

```
Task N 完成标准：
[ ] 代码已运行（不是只写了）
[ ] 输出符合预期（有截图/命令输出为证）
[ ] 测试通过（如果有测试）
[ ] 与 Spec 验收标准对照：<逐条结果>

→ 全部通过后才标记 [x]，才汇报完成
```

---

## 原则

- **声明完成 = 承诺已验证**：完成声明是一种承诺，不是推测
- **证据优先于自信**：再有把握也要跑一遍
- **局部完成 > 虚假完成**：做完一半并如实说，好过声称全做完但没验证
- **跳过验证需要理由**：如果某项检查无法做，说明原因，不能静默跳过
