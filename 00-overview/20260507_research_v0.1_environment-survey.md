## 三、Codex_up 核心设计理念（可直接借鉴）

1. **安全默认**：所有操作默认只读或本地报告，不自动写全局配置、不自动安装、不自动旋转密钥
2. **先调研后规划再实施**：标准工作流是 `doctor --quick → doctor --full → plan → apply`
3. **干跑验证**：所有写操作先 `--dry-run`，审核后 `-Execute`
4. **分层 Rules**：default / read-only / dev-safe / dangerous-block 四层安全规则
5. **诊断统一入口**：`codex_env_doctor.ps1` 聚合所有子检查，输出 ok/warning/error
6. **Memory 纪律**：持久化稳定偏好和工作模式，不持久化 token / 临时任务状态
7. **Skill 演进**：新 Skill 先作为 suggested 草案，经评审后再安装，不自动扩展
8. **进度审计报告**：每轮工作结束必须输出结构化进度报告
9. **可迁移性**：所有路径参数化，脚本支持参数覆盖，避免硬编码

## 四、Claude Code 与 Codex 机制差异对照

| 维度 | Codex (OpenAI) | Claude Code (Anthropic) |
|------|----------------|------------------------|
| 全局配置文件 | `~/.codex/config.toml` | `~/.claude/settings.json` |
| 项目指令文件 | `AGENTS.md` | `CLAUDE.md` |
| 行为规则格式 | `prefix_rule(pattern=..., decision=...)` | `permissions: { allow: [...], deny: [...] }` |
| MCP 配置位置 | `config.toml` 的 `[mcp_servers.*]` 段 | `settings.json` 的 `mcpServers` 字段 |
| 运行时 Profile | config.toml `[profiles.*]` | **无原生 Profile 机制** |
| Hooks 机制 | 无 | **有 Hooks**（pre/post tool 脚本） |
| 自定义命令 | 无 | **有斜杠命令**（`~/.claude/commands/`） |

**关键结论**：Codex_up 是为 OpenAI Codex 构建的，不能直接套用到 Claude Code。但方法论 80% 可以复用。
