## 二、Claude Code 平台能力全景图（2026 现状）

### 2.1 配置体系（三层配置架构）

```
管理层（Managed）      managed-settings.json / MDM / 企业策略
    ↓ 不可被下层覆盖
用户层（User）         ~/.claude/settings.json  （个人全局）
    ↓ 可被项目层覆盖
项目层（Project）      .claude/settings.json   （团队共享，可入 git）
    +
本地层（Local）        .claude/settings.local.json  （个人私有，不入 git）
```

`settings.json` 支持 JSON Schema 验证：`https://json.schemastore.org/claude-code-settings.json`

### 2.2 settings.json 核心字段

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": ["Bash(npm run *)", "Read(**)", "mcp__filesystem__*"],
    "deny": ["Read(**/.env*)", "Read(**/secrets/**)", "Bash(rm -rf *)"]
  },
  "env": {
    "ANTHROPIC_API_KEY": "${env:ANTHROPIC_API_KEY}"
  },
  "hooks": {
    "PreToolUse": [...],
    "PostToolUse": [...],
    "Stop": [...]
  },
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem@latest", "D:\\MoonzWorkspace"]
    }
  },
  "enabledPlugins": ["plugin@marketplace"],
  "autoUpdates": true,
  "verbose": false
}
```

关键位置：
- `~/.claude.json` — OAuth session、MCP 用户级配置、每项目状态（工具信任、设置缓存），**不是** settings.json
- `.mcp.json` — 项目级 MCP servers（project scope）

### 2.3 CLAUDE.md 记忆体系（多层）

```
~/.claude/CLAUDE.md            全局个人偏好（每次对话必读）
./CLAUDE.md                    项目根目录（团队共享，入 git）
./CLAUDE.local.md              项目私有覆盖（不入 git）
./subdir/CLAUDE.md             子目录特定规则（按需加载）
```

设计原则：
- **总长度控制在 200 行以内**（超长会导致 Claude 忽略指令）
- 用 `@path/to/file` 语法导入外部文件，分散内容
- 全局 CLAUDE.md 放稳定偏好，项目 CLAUDE.md 放项目特定指令
- **Auto-memory**：Claude Code 可在会话中自动发现有价值的信息并写回 CLAUDE.md

### 2.4 Hooks 系统（确定性执行保障）

**Hooks 和 CLAUDE.md 的根本区别**：
- CLAUDE.md 是"建议性"的，Claude 可能遵循也可能不遵循
- Hooks 是"确定性"的，每次都执行，无例外

支持的 Hook 事件：
| 事件 | 时机 | 典型用途 |
|------|------|---------|
| `PreToolUse` | 工具调用前 | 验证、阻断危险操作、日志记录 |
| `PostToolUse` | 工具调用后 | 格式化、运行测试、写审计日志 |
| `PostToolUseFailure` | 工具调用失败后 | 错误处理、告警 |
| `PermissionRequest` | 权限请求时 | 动态审批 |
| `Stop` | 对话轮结束 | 写进度摘要、通知 |
| `Notification` | Claude 需要用户输入时 | 桌面通知 |

Hook 配置示例：
```json
"hooks": {
  "PostToolUse": [{
    "matcher": "Write|Edit",
    "hooks": [{
      "type": "command",
      "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\""
    }]
  }],
  "PreToolUse": [{
    "matcher": "Bash",
    "hooks": [{
      "type": "command",
      "command": "echo \"[AUDIT] Bash: $CLAUDE_TOOL_INPUT_COMMAND\" >> ~/.claude/audit.log"
    }]
  }]
}
```

Hook 类型：`command`（Shell）、`http`（POST 到 HTTP 端点）

### 2.5 Skills 系统（可调用能力包）

Skills 是以 SKILL.md 为核心的能力包，格式：
```
skill-name/
  SKILL.md          # 必须，YAML frontmatter + Markdown 指令
  scripts/          # 可选，辅助脚本
  references/       # 可选，参考资料
  agents/           # 可选，子代理定义
```

SKILL.md 格式：
```yaml
---
name: skill-name
description: 一句话描述，决定何时被触发
---
# 技能内容...
```

重要：Skills 是开放标准（2025年12月发布），**Claude Code、Cursor、OpenCode、Gemini CLI、Windsurf 等多工具通用**。

触发方式：
- 用户主动调用：`/skill-name`
- Claude 自动识别触发（基于 description 匹配）

### 2.6 斜杠命令（Slash Commands）

除 Skills 外，还可以在 `.claude/commands/` 放 `.md` 文件创建简单命令：
```
~/.claude/commands/daily-brief.md     # 全局命令 /daily-brief
.claude/commands/review.md            # 项目命令 /review
```

Skills 是 commands 的超集（有 frontmatter + 可携带脚本和子代理）。

### 2.7 MCP（Model Context Protocol）

MCP 配置方式：
```bash
claude mcp add filesystem --scope user npx @modelcontextprotocol/server-filesystem D:\MoonzWorkspace
claude mcp add playwright --scope user npx @playwright/mcp@latest
claude mcp add context7 --scope user npx @upstash/context7-mcp@latest
```

权限控制：
```json

  "allow": ["mcp__filesystem__*"],
  "deny": ["mcp__filesystem__write_file"]
}
```

## 三、社区标杆案例提炼

### 3.1 Karpathy 四原则（行为层）

被社区广泛采纳的编码 Agent 行为准则，适用于开发场景的 CLAUDE.md 全局原则：

1. **Think before coding**（先想后写）：检查现有指令、评估当前状态、确定最小安全改动
2. **Simplicity first**（简单优先）：用现有脚本/模板/直接检查，不增加不必要依赖
3. **Surgical changes**（外科手术式改动）：只改当前任务需要的文件，保留无关的已有内容
4. **Goal-driven execution**（目标驱动）：把工作绑定到明确的计划、阶段或任务，而不是漫无边际

### 3.2 Matt Pocock Skills 哲学（工程流程层）

每个 Skill = 一个可复用的工程动作（不是通用原则，而是具体工作流）：
- `grill-me`：实现前先审问需求，暴露隐藏假设
- `tdd`：测试驱动，垂直切片一次做一个
- `diagnose`：系统化排障流程
- `setup`：理解项目偏好后配置工作环境
- `pr-review`：代码审查标准化
- `git-guardrails`：保护主分支、阻断破坏性 git 操作

核心设计原则：**一个 Skill = 一个原子工程动作**，小而精，不做大而全的 mega-skill。

### 3.3 GenericAgent 架构哲学（记忆与自演化层）

三大设计理念，直接适用于 ClaudeApp 的记忆管理：

1. **上下文信息密度最大化**：保留决策相关信息，而非原始对话历史
2. **层级化按需记忆**：不把所有内容塞入 CLAUDE.md，用索引+按需加载替代全量加载
3. **自演化成可复用流程（SOP）**：重复成功的动作 → 候选技能 → 人工审查 → 全局安装

演化路径：
```
重复执行 → Research 记录（Bottom Line + What To Borrow）
         → 候选 Skill（skills/suggested/）
         → 人工审查
         → 全局安装（~/.claude/skills/ 或 Cowork 插件）
```

### 3.4 研究档案设计（Research Object Archive 范式）

```markdown
# [对象名] Research

## Metadata          # 来源、日期、状态
## Research Question # 明确问题
## Bottom Line       # 一段话结论（压缩精华）
## What To Borrow    # 表格：模式 → 价值 → 适配方式 → 置信度
## Evidence          # 来源 → 类型 → 相关发现 → 链接
## Applicability     # 适用 vs 不适用 + 前提条件
## Risks And Limits  # 四类风险
## Comparison Notes  # 与其他研究对象的横向比较
## Follow-Up         # 行动项（可转化为 Skill 候选）
```

这个范式直接可以作为 ClaudeApp 的 `research` Skill 模板。

## 四、双场景认知框架

**最核心的认知**：ClaudeApp 要服务两种截然不同的使用模式，它们有不同的上下文结构、工具需求、记忆类型和工作节奏。

### 4.1 场景 A：工程开发（Engineering Dev Mode）

**本质**：与代码库对话，驱动确定性的软件生产活动

| 维度 | 特征 |
|------|------|
| 上下文主体 | 代码库、架构、Issue/PR、构建系统 |
| 判断标准 | 代码跑通、测试通过、类型检查 ok |
| 安全边界 | 主分支保护、禁止 rm -rf、禁止 git reset --hard |
| 记忆内容 | 构建命令、代码风格、目录结构、常见错误解法 |
| MCP 核心 | filesystem、playwright（E2E）、context7（文档）、chrome-devtools |
| Hooks 核心 | PostToolUse → 自动格式化 + 运行测试；PreToolUse → Bash 审计日志 |
| Skills 核心 | tdd、diagnose、grill-plan、pr-review、architecture-review、security-audit |
| CLAUDE.md 重点 | 仓库结构、构建命令、测试框架、代码规范、禁止操作列表 |
| 工作节奏 | 短循环（改一处 → 验证 → 继续），可长程自主 |

**核心原则（开发场景）**：
1. 代码优先于指令：让构建和测试说话，不要只听文字描述
2. 最小可验证增量：每次改动都要有对应的验证方式
3. 安全边界明确：危险操作清单写进 permissions.deny，不依赖 CLAUDE.md 提醒
4. 工具链统一：在 CLAUDE.md 里说明用什么测试框架、格式化工具，避免每次重新确认
5. Skill 为工程动作建模：把 TDD、diagnose、PR review 等固化为可复用的 Skill

### 4.2 场景 B：个人数字助理（Personal Digital Assistant Mode）

**本质**：与知识、任务、内容对话，支持思考、决策和信息处理

| 维度 | 特征 |
|------|------|
| 上下文主体 | 任务列表、会议记录、研究材料、文档、个人决策 |
| 判断标准 | 信息准确、格式符合习惯、节省时间 |
| 安全边界 | 不写生产系统、不发送邮件/消息（需确认）、不修改日历（需确认）|
| 记忆内容 | 工作角色、核心项目列表、写作偏好、决策风格、领域词汇表 |
| MCP 核心 | filesystem（本地文档）、web search、notion/飞书（如有）、calendar（只读）|
| Hooks 核心 | Stop → 自动写会话摘要；PostToolUse → 写入研究档案 |
| Skills 核心 | summarize-meeting、research-object-archive、daily-brief、weekly-review、create-prd、decision-log |
| CLAUDE.md 重点 | 角色定义、沟通风格、输出语言、领域术语、常用模板路径 |
| 工作节奏 | 中长循环（研究 → 整理 → 沉淀），有时需要跨会话的连续性 |

**核心原则（助理场景）**：
1. 个性化优先于通用性：CLAUDE.md 要告诉 Claude 你是谁、你的词汇表、你的决策风格
2. 记忆纪律：持久化稳定偏好和可复用模板，不持久化临时状态和对话噪音
3. 输出可沉淀：每次有价值的输出都应该有落地路径（Research 档案、模板、决策日志）
4. 工具连接而非工具依赖：MCP 扩展能力，但不让工具代替思考
5. 信任边界清晰：发送操作（邮件、消息）必须经过确认，永远不自动执行

### 4.3 双场景的共通原则

无论开发还是助理，以下原则贯穿始终：

| 原则 | 含义 |
|------|------|
| **配置先于行动** | 先固化 settings.json + CLAUDE.md，再开始使用 |
| **安全默认** | 默认只读或本地操作，写入和外部调用需要明确授权 |
| **信息密度管理** | CLAUDE.md < 200 行；研究档案用 Bottom Line 压缩 |
| **演化而非堆砌** | 新能力先作为候选（suggested/），验证后再安装 |
| **可验证的进度** | 每轮工作结束要有结构化报告：做了什么、验证了什么、下一步什么 |
| **密钥永不在文档里** | 用 `${env:VAR}` 语法引用，永远不硬编码 |
| **干跑优先** | 任何批量写入、安装、迁移都先 dry-run 审核 |

## 八、本轮调研总结

### 关键认知沉淀

1. **Claude_up 是个人环境治理包，不是工具堆砌**：核心价值在配置可复现、能力可演化、记忆有纪律
2. **双场景设计是核心差异**：开发场景需要代码确定性，助理场景需要知识密度
3. **三层能力分离**：CLAUDE.md（建议性原则）+ settings.json（强制边界）+ Hooks（确定性行为）各司其职
4. **Skills 要小而精**：一个 Skill 对应一个原子工作流，不做 mega-skill
5. **研究档案范式已经是 GenericAgent 思想的落地**：高密度信息保存 + 演化成可复用工具
6. **Codex_up 的方法论可以 80% 复用**：目录哲学、安全模型、诊断模式、演化纪律直接迁移
